> #### Contexto / Problema
> O fluxo de cadastro do adolescente já possui uma tela intermediária (`GET /cadastro/adolescente/escolher-metodo`) que exibe os métodos de autenticação (Google, Apple desativados + e-mail/senha). ## 1. Contexto: estado atual do sistema

##### 1.1. Resumo da área da demanda
O módulo `accounts` gerencia o cadastro de responsável (US-003) e adolescente (US-001). O fluxo do adolescente já possui uma tela intermediária de escolha de método (`/cadastro/adolescente/escolher-metodo`) com botões sociais desabilitados + link para e-mail/senha. O fluxo do responsável pula diretamente para o form de e-mail/senha (`/cadastro/responsavel`), sem tela intermediária. A página de escolha de papel (`/cadastro`) direciona o card "Sou responsável" diretamente para `/cadastro/responsavel`.

## 2. Objetivo desta task

##### 2.1. Resultado funcional alvo
Criar uma tela intermediária de escolha de método de autenticação no fluxo de cadastro do responsável, espelhando a tela existente para adolescentes. O card "Sou responsável" na página de escolha de papel passa a apontar para esta nova tela.

##### 2.2. Surface comportamental coberta
- Usuário acessa `/cadastro/responsavel/escolher-metodo` e vê botões Google/Apple desabilitados + link ativo para e-mail/senha
- Botão voltar na nova tela retorna à página de escolha de papel (`/cadastro`)
- Microcopy da nova tela é específico para responsável (não copia texto do adolescente)

##### 2.3. Constraints do produto / demanda
- Reutilizar classes CSS existentes (`escolher-metodo`, `overline`, `divider`, `btn-email`, `note`) — não criar CSS novo
- Reutilizar fragmento `social-methods` existente (não editar)
- Usar `layout/public` (não `base`)
- Botão voltar via fragmento `public-header(backHref, backLabel)` apontando para `/cadastro`
- Constraints LGPD: N/A — sem superfície de dados pessoais (tela estática com botões desabilitados + link)

## 4. Observações factuais (gotchas)

- **Test framework:** a issue menciona `WebTestClient` nos testes, mas o código existente usa **MockMvc** (`org.springframework.test.web.servlet.MockMvc`) com `@SpringBootTest` + Testcontainers (Postgres). Os testes existentes em `GuardianRegistrationControllerIT.java` e `SignupEntryControllerIT.java` usam `MvcResult`, `andExpect()`, `containsString()` — não `WebTestClient`.
- **Naming de métodos de teste:** convenção existente é camelCase descritivo (ex.: `getRegistrationFormReturnsView()`, `postValidDataRedirectsToVincular()`), não `shouldXxxWhenYyy`.
- **Layout:** a tela do adolescente usa `layout:decorate="~{layout/public}"` com `<th:block layout:fragment="content">`. O `public-header` fragment recebe `(backHref, backLabel)` como parâmetros.
- **CSS:** as classes `.escolher-metodo`, `.overline`, `.divider`, `.btn-email`, `.note` já existem no `app.css` (usadas pela tela do adolescente).
- **Fragmento social-methods:** já renderiza botões Google/Apple com `disabled` e `aria-disabled="true"`. Não precisa ser editado.
- **Discrepância catálogo vs issue:** `doc/Requisitos/UserStory.md` lista US-004 como "Cadastro de responsável via Google" (OAuth), enquanto a Issue #100 descreve US-004 como "Tela de escolha de método de autenticação do responsável" (UI). O scout captura o que está na Issue #100.
- **Sem spec formal:** não existe `doc/UX/us-004-spec.md`. Apenas protótipo estático em `doc/UX/prototypes/cadastro-responsavel-metodo.html`.
