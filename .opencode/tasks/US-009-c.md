# US-009-c · Photo upload + Action sheet + Testes

## Objetivo

Completar a edição do perfil com: (1) action sheet bottom no template para trocar/remover foto; (2) integração multipart no controller `ProfileController` e service `UpdateProfileService` usando `AvatarStorage.store()`; (3) remoção de foto via `avatarUrl = null`; (4) MockMvc tests cobrindo save foto, remove foto e rejeição de arquivo inválido.

Depende de: US-009-a (controller + service base), US-009-b (template com avatar block).

## Contexto

| Arquivo | Ação |

### Padrão escolhido

Seguir o padrão de multipart já consolidado em `AdolescentRegistrationController` (`@RequestParam(value = "photo", required = false) MultipartFile photo`) + `RegisterAdolescentService`. O service delega a `AvatarStorage.store()` (interface existente com implementação `FilesystemAvatarStorage`). Exceções `AvatarTooLargeException` e `AvatarUnsupportedTypeException` são capturadas no controller e transformadas em flash message de erro.

No template, Alpine.js gerencia o action sheet (`x-data="{ showSheet: false }`) e os dois `<input type="file">` ocultos (um com `capture="user"` para câmera frontal, outro sem para galeria). Ao selecionar um arquivo, o form é submetido via JavaScript nativo (`form.submit()`) — não HTMX.

### Alternativas descartadas

1. **HTMX para upload de foto** — rejeitado: HTMX `hx-encode` com multipart funciona, mas o projeto ainda não usa HTMX para forms file; o padrão existente (`AdolescentRegistrationController`) usa submit nativo + redirect. Manter consistência.
2. **Endpoint dedicado `POST /perfil/foto`** — rejeitado: a US-009 trata edição do perfil como um todo (nickname + birthDate + foto); separar em outro endpoint duplicaria a lógica de resolução do accountId e validação de role. O POST `/perfil` existente já aceita form; basta adicionar o param multipart opcional.
3. **Deletar arquivo antigo do filesystem ao trocar foto** — rejeitado: o brief (§4) explicita que remoção é só `avatarUrl = null`; limpeza de órfãos fica para uma chore futura.
