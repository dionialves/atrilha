**`FilesystemAvatarStorage`** — implementação existente de `AvatarStorage`:
```java
// src/main/java/dev/zayt/atrilha/accounts/avatar/FilesystemAvatarStorage.java (linhas 27-73)
@Component
public final class FilesystemAvatarStorage implements AvatarStorage {
    private static final long MAX_BYTES = 5L * 1024L * 1024L;
    private static final Map<String, String> EXTENSION_BY_MIME = Map.of(
            "image/jpeg", "jpg", "image/png", "png", "image/webp", "webp");

    @Override
    public String store(UUID accountId, MultipartFile file) {
        if (file == null || file.isEmpty()) throw new AvatarUnsupportedTypeException("vazio");
        String contentType = file.getContentType();
        String extension = EXTENSION_BY_MIME.get(contentType == null ? "" : contentType);
        if (extension == null) throw new AvatarUnsupportedTypeException(contentType);
        long size = file.getSize();
        if (size > MAX_BYTES) throw new AvatarTooLargeException(size, MAX_BYTES);
        Path destDir = baseDir.resolve("avatars");
        Path destFile = destDir.resolve(accountId + "." + extension);
        try (var in = file.getInputStream()) {
            Files.copy(in, destFile, StandardCopyOption.REPLACE_EXISTING);
        } catch (IOException e) { throw new UncheckedIOException("Falha ao gravar avatar", e); }
        return "/media/avatars/" + accountId + "." + extension;
    }
}
```

## 2. Objetivo desta demanda
#### 2.1. Resultado funcional alvo (desta subtask)
Action sheet bottom abre ao clicar no badge de câmera do avatar. Opções: "Tirar foto agora" (câmera nativa), "Escolher da galeria", "Tirar foto (ficar com a inicial)" — remove `avatarUrl`. Controller aceita multipart photo no POST; service chama `AvatarStorage.store()` para salvar ou seta `avatarUrl = null` para remover. Tests cobrem save foto, remove foto, rejeita arquivo inválido.

#### 2.2. Surface comportamental coberta
- **Action sheet:** abre ao clicar no badge de câmera do avatar; 4 opções (câmera, galeria, remover, cancelar); fecha com Escape ou click no overlay
- **Upload de foto:** `<input type="file" accept="image/*">` oculto; ao selecionar, submete POST `/perfil` com multipart photo
- **Remoção de foto:** botão "Tirar foto (ficar com a inicial)" no sheet → POST `/perfil` sem photo → service seta `avatarUrl = null`
- **Rejeição de arquivo inválido:** `AvatarTooLargeException` ou `AvatarUnsupportedTypeException` → banner error na tela

#### 2.3. Constraints do produto/demanda + constraints do AGENTS.md
- **Stack travada:** SIM/NÃO — `FilesystemAvatarStorage` já existe (ADR-010); reusar sem criar nova implementação
- **Compliance LGPD:** SIM — remoção de foto não apaga outros dados do perfil

## 4. Observações factuais (gotchas)
- `FilesystemAvatarStorage.store()` já retorna URL relativa `/media/avatars/{accountId}.{ext}` — o service só precisa chamar e salvar no perfil.
- A remoção de foto é simples: setar `profile.setAvatarUrl(null)` e salvar. Não precisa deletar o arquivo do filesystem (o controller pode limpar arquivos órfãos em uma chore futura).
- O spec UX (§3.7) define que "Tirar foto (ficar com a inicial)" só aparece quando há foto setada — condicional no template: `x-show="avatarUrl"`.
- `AvatarTooLargeException` e `AvatarUnsupportedTypeException` são exceções dedicadas já definidas — o controller deve capturar e transformar em erro de binding ou flash message.
