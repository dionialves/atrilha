```java
Optional<AgeEligibilityViolation> check(LocalDate birthDate, AccountRole role) {
    int age = Period.between(birthDate, today).getYears();
    return switch (role) {
        case TEEN -> { if (age < 13) yield violation(YOUNG); }
                     { if (age >= 18) yield violation(OLD); }
                     yield Optional.empty(); }
    };
}
```

## 2. Objetivo desta demanda
#### 2.1. Resultado funcional alvo (desta subtask)
Controller `GET /perfil` retorna modelo com dados do perfil (nickname, birthDate, email, avatarUrl). Controller `POST /perfil` valida e atualiza nickname + birthDate via service. Validação server-side: nickname 3–20 chars, birthDate dentro de 13–17 anos (via `AgeEligibilityChecker`). MockMvc tests cobrem happy path, erros de validação e bloqueio por idade. SecurityConfig protege `/perfil/**` com `ROLE_TEEN`.

#### 2.2. Surface comportamental coberta
- `GET /perfil` → retorna modelo com dados do perfil (view é implementada em US-009-b)
- `POST /perfil` → valida nickname (3–20 chars), birthDate (idade 13–17); atualiza no banco; retorna redirect com flash message de sucesso ou re-renderiza com erros
- Validação server-side via Jakarta Bean Validation + `AgeEligibilityChecker`

#### 2.3. Constraints do produto/demanda + constraints do AGENTS.md
- **Stack travada:** SIM/NÃO — Java 21 + Spring Boot 4 + Spring Security (tudo dentro do stack aprovado)
- **Compliance LGPD:** SIM — edição não pode apagar progresso/vinculação. E-mail é readonly (não editado aqui).

## 4. Observações factuais (gotchas)
- O campo `email` está na tabela `accounts`, não em `adolescent_profiles`. O controller precisa fazer lookup separado (`AccountRepository.findByEmailIgnoreCaseAndDeletedAtIsNull`) para exibir o e-mail no modelo GET.
- `AgeEligibilityChecker.check()` retorna `TEEN_TOO_OLD` quando age ≥ 18 — este é o gatilho de erro no POST. A view (US-009-b) traduz isso em modal, não o controller.
- O padrão de autenticação dual (`AuthenticatedAccount` pós-cadastro + `AtrilhaUserDetails` via form login) exige que o controller resolva o accountId a partir do SecurityContext:
```java
// Padrão existente em PostLoginRedirectController (linhas 28-44)
Object principal = authentication.getPrincipal();
if (principal instanceof AuthenticatedAccount acc) {
    // Fluxo de cadastro: principal é AuthenticatedAccount (UUID + role).
} else if (principal instanceof AuthenticatedPrincipal ap) {
    // Fluxo de form login: principal é AtrilhaUserDetails.
}
```
- `SecurityConfig.java` usa `.requestMatchers("/trilha/**").hasRole("TEEN")` como padrão — seguir o mesmo estilo para `/perfil/**`.
